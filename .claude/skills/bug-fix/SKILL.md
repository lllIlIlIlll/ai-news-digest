# 通用 Bug 修复流程规范

基于《CLAUDE.md》中"目标驱动执行"原则，适用于任何技术栈和问题类型。确保每次修复都有明确目标和可验证标准。

## 核心原则

**先定义成功，再实施修复。** 修复完成后必须能明确回答："这个 bug 真的修好了吗？"

## 修复流程

### 1. 问题分析阶段
- **现象复现**：重现 bug 的具体表现
- **根因定位**：确定 bug 的直接原因和深层原因
- **影响范围**：评估影响的功能和代码模块

### 2. 验收标准建立（修复前必须完成）
> 回答这两个问题，形成可验证的验收标准

#### 2.1 必须消失的现象
**问题**：修复完成后，什么现象必须消失？
- 描述 bug 的具体表现（错误消息、异常行为、错误输出等）
- 这些现象在修复后应该完全消失

**示例**：
- ❌ 错误："TypeError: Cannot read property 'title' of undefined"
- ❌ 现象：当 RSS 源返回空文章列表时程序崩溃
- ✅ 验收：程序能正确处理空文章列表，不崩溃，返回空数组

#### 2.2 必须出现的结果
**问题**：什么结果出现了，才算修复成功？
- 描述修复后系统的正确行为
- 包括正常情况和边界情况的预期结果

**示例**：
- ✅ 结果：传入空数组时返回空数组，不抛出异常
- ✅ 结果：控制台显示"成功处理 0 篇文章"
- ✅ 结果：后续处理流程能正常继续

### 3. 验证方法设计
如果适合，将验收标准转化为自动检查或测试：

#### 3.1 测试用例设计
```typescript
// 示例：基于上述验收标准的测试
it('应该正确处理空文章列表', async () => {
  // 重现 bug 的场景
  const emptyArticles: Article[] = [];
  
  // 调用被修复的函数
  const result = await processArticles(emptyArticles);
  
  // 验证"必须出现的结果"
  expect(result).toEqual([]);
  expect(consoleSpy).toHaveBeenCalledWith('成功处理 0 篇文章');
  
  // 验证"必须消失的现象" - 不应该抛出异常
  expect(() => processArticles(emptyArticles)).not.toThrow();
});
```

#### 3.2 手动验证清单
如果无法自动化：
```
- [ ] 重现 bug 的步骤不再导致错误
- [ ] 正常功能不受影响
- [ ] 边界情况处理正确
- [ ] 错误日志中不再出现相关错误
```

### 4. 修复实施
**约束**：以通过验收标准为目标，最小化修改

#### 4.1 修改策略
- 只修改必需代码，不重构无关部分
- 遵循项目现有代码风格
- 保留原有测试，仅添加新测试验证修复

#### 4.2 代码审查清单
```
- [ ] 修改直接针对根因
- [ ] 没有引入新问题
- [ ] 遵循 TypeScript 核心规则
- [ ] 保持向后兼容性（如需要）
```

### 5. 验证执行
修复完成后必须执行验证：

#### 5.1 运行验证
```bash
# 运行相关测试
bun test tests/path/to/fix.test.ts

# 或运行完整测试套件
bun test

# 或手动执行验证步骤
bun run start --test-bug-scenario
```

#### 5.2 验证结果
- **通过**：所有验收标准满足，测试通过
- **失败**：回到步骤1重新分析
- **部分通过**：调整验收标准或修复方案

## 示例：修复 TypeScript 类型安全违规

### 问题现象
在 `src/utils/html.ts` 中发现 `any` 类型使用，违反 TypeScript 核心规则。

### 验收标准建立
1. **必须消失的现象**：
   - ❌ 文件中不再有 `: any` 类型注解
   - ❌ 不再有 `as any` 类型断言（第三方库交互除外）
   - ❌ TypeScript 编译时不再有相关类型警告

2. **必须出现的结果**：
   - ✅ 所有变量和函数都有具体类型定义
   - ✅ 使用正确的接口或类型别名
   - ✅ 代码通过严格模式 TypeScript 编译
   - ✅ 现有测试仍然通过

### 验证方法
```bash
# 自动检查
npx tsc --noEmit --strict

# 运行测试
bun test tests/unit/html.test.ts

# 特定检查
grep -r ": any" src/  # 应该返回空
grep -r "as any" src/ # 应该返回空（除第三方库交互）
```

### 修复实施
1. 分析每个 `any` 的使用场景
2. 定义正确的接口或类型
3. 替换 `any` 为具体类型
4. 确保类型安全

### 验证执行
```bash
# 运行验证
bun test
npx tsc --noEmit
grep -r ": any" src/ | wc -l  # 应该为0
```

## 注意事项

1. **不要一上来就大改代码**：先建立验收标准，再实施修复
2. **不要只说"已经修好了"**：必须提供验证证据
3. **保持最小修改**：只修复问题，不进行无关优化
4. **文档更新**：如果 bug 有普遍性，更新 `docs/errors.md` 和 `CLAUDE.md`
